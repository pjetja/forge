# Phase 01.3: Figma UI Library - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Create a Figma component library for the Forge design system — covering the dark navy tokens, components, and patterns established in Phase 01.1 and 01.2 — plus Phase 2 (Exercise Library) UI patterns designed ahead of implementation. This is a solo reference file used to design future screens before coding. Screens (full page compositions) are NOT in scope — component library only.

</domain>

<decisions>
## Implementation Decisions

### Purpose & Output
- Deliverable: a single Figma file (free tier — 1 of 3 drafts)
- Audience: solo reference for the developer/designer
- Primary use: designing future phases' screens in Figma before coding
- The Figma file must be shareable (link stored in docs/FIGMA-WORKFLOW.md)
- Component library only — no full-page screen compositions
- Scope = existing design system (Phase 01.1/01.2 tokens + components) + universal atoms + Phase 2 (Exercise Library) patterns
- Future phases each get their own UI sub-phase (02.1, 03.1, 04.1, 05.1) for phase-specific Figma design work

### Creation Method
- Primary approach: Figma AI (Make Designs) to bootstrap components — but free-tier AI availability must be researched first
- Fallback: Figma plugin script (JS) that Claude writes to programmatically create base structure (pages, token frames, placeholder components)
- Plans use mixed execution: code tasks (Claude executes) + manual Figma tasks (user executes with human-verify checkpoints)

### Figma MCP
- Research and set up Figma MCP as the FIRST task in this phase
- Goal: minimize manual Figma work by letting Claude write directly to Figma via MCP
- If MCP setup succeeds → subsequent tasks use MCP for component creation
- If MCP unavailable → fall back to plugin script + manual approach

### Component Scope
Core (existing design system):
- Design tokens: colors (navy palette, emerald, grays), typography (Lato scale + weights), spacing, border-radius, shadows
- Buttons & interactive: Primary, Secondary, Ghost, Icon button variants; loading states; disabled state
- Form components: Text input, Textarea, Select, Checkbox, Radio button, Label, Error state — all with Default/Focus/Error/Disabled variants
- Layout & navigation: Trainer nav shell, Trainee nav shell, Sidebar pattern, Page header structure
- ForgeLogo: both variants (Horizontal for nav, Icon-only for favicon/app icon)
- Modals & dialogs: Generic modal shell + InviteDialog + generic Confirm dialog
- Data display: Card/list item template, Table with header/rows, Badges/tags, Empty state (illustration placeholder + copy)

Phase 2 patterns (designed ahead of implementation):
- Exercise/Card — exercise name, muscle group tag, description preview
- Exercise/SearchBar — search input + muscle group filter chips
- Exercise/CreateForm — multi-field form (name, muscle group, description)
- Exercise/DetailView — full exercise display with all fields

### Component Fidelity
- Functional states only: Default, Disabled, Error, Loading — not exhaustive Figma variant matrices
- Full Figma components (not flat frames) — main components on library pages, instances on design/phase pages
- Within free tier single-file constraint, Figma component system enables update-once-reflects-everywhere

### Library Structure (Atomic Design)
Pages in order:
1. Tokens — color swatches (left) + typography scale (right), side by side
2. Atoms — Buttons, Inputs, Badges, Logo, single-purpose elements
3. Molecules — Form groups, Search bars, Card shells, Modal shell
4. Organisms — Nav shells, Table with data, Empty state patterns
5. Phase 2: Exercise Library — Exercise/Card, Exercise/SearchBar, Exercise/CreateForm, Exercise/DetailView

No cover page — start directly with Tokens.

Layout within pages: Component name as section header label, variants displayed in a row beneath it.

### Component Naming
- Figma-native slash convention: Category/ComponentName (e.g. Buttons/Primary, Inputs/Text, Navigation/Sidebar)
- Variant names are semantic (Primary, Secondary, Accent, Muted, Danger) — not Tailwind class names
- Phase 2 components use feature group naming (Exercise/Card) — no phase prefix

### Code Sync Strategy
- Manual sync — design system is largely stable after Phase 01.1; manual Figma updates when tokens change
- docs/design-tokens.md: Claude generates this file from globals.css @theme — structured token reference for manual Figma updates
- docs/FIGMA-WORKFLOW.md: Claude writes this — documents how to use the library for future sub-phases, when to update tokens, naming conventions, Figma link
- Single Figma file for all content (free tier: 3-file limit)

### Code Tasks (Claude executes in this phase)
1. Research Figma MCP — investigate available MCP servers, set up if feasible
2. Generate docs/design-tokens.md — extract all CSS custom properties from globals.css
3. Write docs/FIGMA-WORKFLOW.md — design workflow, Figma link placeholder, sub-phase process
4. Write Figma plugin script (docs/figma-bootstrap.js) — JS that creates base page structure in Figma
5. Update ROADMAP.md — fill in [Urgent work - to be planned] goals for sub-phases 02.1, 03.1, 04.1, 05.1

### Done Criteria
Phase is complete when ALL of the following are true:
- Figma file exists with a shareable link recorded in docs/FIGMA-WORKFLOW.md
- All planned components built in Figma (tokens, atoms, molecules, organisms, Phase 2 patterns)
- docs/design-tokens.md committed to repo
- docs/FIGMA-WORKFLOW.md committed with Figma link filled in
- Human-verify checkpoint passed: user visually reviews Figma file and approves

### Claude's Discretion

- Which Figma MCP server to use (if multiple options exist)
- Specific illustration choices for empty state placeholders
- Token swatch layout details within the Tokens page
- Spacing between section headers and component rows
