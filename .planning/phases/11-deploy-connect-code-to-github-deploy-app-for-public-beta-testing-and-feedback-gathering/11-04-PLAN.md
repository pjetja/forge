<tasks>

<task type="human_action">
  <name>Task 1: Configure Supabase Auth redirect URLs for production</name>
  <what-needed>
    Supabase enforces a whitelist for `emailRedirectTo` URLs. Without this, signup confirmation links silently redirect to the wrong URL.

    1. Go to your Supabase project dashboard: https://supabase.com/dashboard
    2. Select your Forge project
    3. Navigate to **Authentication → URL Configuration**
    4. Update **Site URL**:
       - Change `http://localhost:3000` to `https://[your-app].vercel.app`
    5. Under **Redirect URLs**, add:
       - `https://[your-app].vercel.app/**`
       - Keep `http://localhost:3000/**` (needed for local dev)
    6. Click **Save**

    Replace `[your-app]` with your actual Vercel subdomain from Plan 11-03.
  </what-needed>
  <resume-signal>Type "supabase-configured" once Site URL and Redirect URLs are saved</resume-signal>
</task>

<task type="human_action">
  <name>Task 2: Run demo seed against production Supabase</name>
  <what-needed>
    Your local `.env.local` should already contain the production Supabase credentials (`NEXT_PUBLIC_SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`). The seed script uses these to connect directly to production.

    In your terminal at the workspace root:
    ```bash
    pnpm seed:demo
    ```

    **Expected output on success:**
    ```
    ✓ Created demo trainer: demo.trainer@forge.app
    ✓ Created demo trainee: demo.trainee@forge.app
    ✓ Demo seed complete
    ```

    If you see an error:
    - `SUPABASE_SERVICE_ROLE_KEY not set` → check `.env.local` has the service role key
    - `DEMO_TRAINER_EMAIL not set` → add demo user env vars to `.env.local` (see `.env.local.example`)
    - Supabase auth error → verify your service_role key matches the production project
  </what-needed>
  <resume-signal>Type "seeded" once pnpm seed:demo completes successfully</resume-signal>
</task>

<task type="auto">
  <name>Task 3: Verify demo seed idempotency</name>
  <files></files>
  <action>
    Run `pnpm seed:demo` a second time to confirm the idempotency guard works. The script should detect the existing demo trainer in the `trainers` table and exit gracefully without creating duplicate records.

    ```bash
    pnpm seed:demo
    ```

    Expected output: `✓ Demo users already seeded — nothing to do.` (or equivalent message)

    If the script errors or creates duplicate records, the seed script has a bug in its idempotency check — investigate `scripts/seed-demo.ts` line that checks for existing trainer.
  </action>
  <verify>
    <automated>pnpm seed:demo 2>&1 | grep -i "already\|seeded\|nothing to do"</automated>
  </verify>
  <done>Running pnpm seed:demo twice exits 0 on the second run with an "already seeded" message</done>
</task>
