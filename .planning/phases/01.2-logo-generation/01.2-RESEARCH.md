## Summary

This phase replaces the `⚡ Forge` emoji placeholder in both app layouts with a designed SVG logo. Two distinct concerns exist: (1) the in-app logo component used in the nav and on auth pages, and (2) the favicon file used in the browser tab.

For in-app use, the cleanest pattern in this codebase is a hand-authored SVG rendered directly as a React component (`.tsx` file returning `<svg>`). No SVGR or webpack SVG loader is needed — the project already uses inline SVGs for the sign-out icon in both layouts, establishing this as the house style. A single `ForgeLogo` component with a `variant` prop (`"horizontal" | "icon"`) handles both shapes from one file.

For the favicon, Next.js App Router's file-based metadata system supports `icon.svg` placed in `src/app/` directly — no metadata export required. An `.ico` fallback for older browsers is retained by keeping the existing `favicon.ico` and adding the SVG `icon.svg` alongside it. The existing `metadata` export in `src/app/layout.tsx` already sets `title: 'Forge'` — no change needed there.

**Primary recommendation:** Author the logo as a single `src/components/ForgeLogo.tsx` component with `variant` prop. Place `icon.svg` in `src/app/` for the SVG favicon. Replace the `⚡ Forge` `<Link>` in both layouts inline.

### Pattern: Single Component, Variant Prop

```tsx
// src/components/ForgeLogo.tsx
// Source: project convention (inline SVG, matches sign-out icon in layouts)

interface ForgeLogoProps {
  variant?: 'horizontal' | 'icon';
  className?: string;
}

export function ForgeLogo({ variant = 'horizontal', className }: ForgeLogoProps) {
  if (variant === 'icon') {
    return (
      <svg
        xmlns="http://www.w3.org/2000/svg"
        viewBox="0 0 32 32"
        className={className}
        aria-label="Forge"
        role="img"
      >
        {/* icon-only paths here */}
      </svg>
    );
  }

  return (
    <svg
      xmlns="http://www.w3.org/2000/svg"
      viewBox="0 0 120 32"
      className={className}
      aria-label="Forge"
      role="img"
    >
      {/* icon + wordmark paths here */}
    </svg>
  );
}
```

**Why one file, not two separate components:**
- The icon shape is shared between variants — changes to the mark stay in sync
- The planner can generate both with one task
- Import path is predictable: `@/components/ForgeLogo`

### Pattern: SVG Favicon via File Convention

```
src/app/icon.svg   ← Next.js auto-generates: <link rel="icon" href="/icon.svg" sizes="any" type="image/svg+xml" />
src/app/favicon.ico  ← kept as ICO fallback for legacy browsers
```

No code changes needed in `layout.tsx` for the favicon — file presence is sufficient.

### Anti-Patterns to Avoid
- **Using `fill="currentColor"` in the wordmark text**: The wordmark color is fixed (`#e2e8f0` or white), not inherited from parent text color. Reserve `currentColor` for purely monochrome icons.
- **Setting explicit `width`/`height` attributes on the SVG root**: Use `className="h-7 w-auto"` with `viewBox` only — this makes the logo scale correctly across all placements.
- **Separate `ForgeIconOnly.tsx` and `ForgeHorizontal.tsx` files**: Single component with variant prop is easier to maintain and import.
- **Removing `favicon.ico`**: Some browsers (especially embedded WebViews) still request `favicon.ico` directly. Keep it as fallback even when `icon.svg` is present.

## SVG + Tailwind CSS Custom Properties: Key Facts

**Question 5 from the brief answered definitively:**

SVG elements rendered as React components (inline in JSX) fully support CSS custom properties, including Tailwind v4's `--color-*` variables. Three approaches work:

1. **Tailwind fill utility class** — `className="fill-accent"` maps to `fill: var(--color-accent)` (Tailwind v4 resolves design token names to CSS variables). This works because this project defines `--color-accent: #10b981` in `@theme inline`.

2. **Arbitrary fill class** — `fill-[#10b981]` — hard-codes the hex, bypasses the token system. Fine for logo brand colors that won't change.

3. **SVG `fill` attribute** — `fill="#10b981"` directly on the `<path>` — most explicit, not affected by Tailwind at all. Recommended for logo paths where the color is a brand constant.

**Recommendation for this phase:** Use hard-coded hex values (`fill="#10b981"` for emerald, `fill="#e2e8f0"` for wordmark text) directly in the SVG `<path>` attributes. The logo is brand identity, not a theme-switchable component. This avoids any CSS class specificity issues and makes the SVG file reusable outside of React (e.g., for `icon.svg` favicon).

## Common Pitfalls

### Pitfall 1: SVG favicon not rendering in some browsers
**What goes wrong:** `icon.svg` is served but some browsers (Firefox older versions, Safari) show a blank icon.
**Why it happens:** SVG favicons require `type="image/svg+xml"` in the `<link>` tag. Next.js adds `sizes="any"` for SVG but the MIME type must be correct.
**How to avoid:** Next.js file convention for `icon.svg` sets the correct MIME type automatically. Verify by inspecting `<head>` output in dev.
**Warning signs:** Blank favicon in browser tab after deployment but not locally (caching).

### Pitfall 2: `viewBox` missing causes SVG to not scale
**What goes wrong:** SVG appears either fixed-size or invisible.
**Why it happens:** Without `viewBox`, the SVG has no intrinsic coordinate system and `width: auto` has no reference point.
**How to avoid:** Always set `viewBox="0 0 W H"`. Never set fixed `width`/`height` attributes on the root `<svg>` — let Tailwind classes control sizing.

### Pitfall 3: Logo Link loses accessible label when text is removed
**What goes wrong:** Screen readers announce nothing when the `⚡ Forge` text is replaced with a graphic SVG.
**Why it happens:** The `<Link>` content was previously text; now it is an image.
**How to avoid:** Add `aria-label="Forge home"` on the `<Link>` or add `aria-label="Forge"` + `role="img"` on the `<svg>`. Use both for belt-and-suspenders.

### Pitfall 4: Auth layout centering breaks with logo above card
**What goes wrong:** Logo and card appear misaligned or logo is outside the centered column.
**Why it happens:** `AuthLayout` renders `<div className="w-full max-w-md px-4">{children}</div>` — if the page component returns a fragment, both elements inherit this `max-w-md` constraint, which is correct. But if wrapped in an outer div with its own flex/centering, it can double-center.
**How to avoid:** Return a React fragment (`<>...</>`) or a plain `<div className="space-y-6">` from the page — no additional centering wrapper. The auth layout already handles horizontal centering.

### Pitfall 5: `favicon.ico` in `public/` vs `src/app/`
**What goes wrong:** Developer places `icon.svg` in `public/` expecting it to work as a favicon.
**Why it happens:** Confusion between static file serving (`public/`) and metadata file conventions (`src/app/`).
**How to avoid:** The file-based favicon convention requires the file to be in the `app/` directory (i.e., `src/app/icon.svg`), not `public/`. The existing `favicon.ico` at `src/app/favicon.ico` is already in the right place — `icon.svg` goes there too.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| SVGR webpack loader for SVG-as-component | Inline SVG in TSX (no loader) | Next.js 13+ App Router era | Simpler, no config, works with Turbopack |
| `<Head>` component for favicon | File-based `icon.svg` in `app/` | Next.js 13.3 | Zero-code favicon; metadata API handles `<head>` injection |
| Separate `.ico` + multiple PNG sizes (512, 192, 32, 16) | `icon.svg` + one `.ico` fallback | 2021+ (browsers caught up) | SVG scales infinitely; only need one fallback ICO |

**Deprecated/outdated:**
- `pages/_document.tsx` with custom `<Head>` for favicon: not used in App Router
- `next/head` import for metadata: replaced by `export const metadata` and file conventions
