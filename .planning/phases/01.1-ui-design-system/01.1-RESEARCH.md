## Summary

This phase is a pure visual rework of the existing Forge app: replace the current light (white/gray) theme with a dark navy design system using emerald green accents and Lato typography. No new features or routes — only the styling of 8 existing pages and 2 app shell layouts changes.

The project uses **Tailwind CSS v4** (via `@tailwindcss/postcss`) and **Next.js 16.1.6** (App Router). Tailwind v4 replaces `tailwind.config.js` with a CSS-first `@theme` block in `globals.css`. All custom color tokens, font tokens, and design variables live in `globals.css` — this is the single source of truth. The `next/font` module loads Lato at build time (self-hosted, no FOUC), exposes it as a CSS variable, and that variable is mapped into Tailwind's `--font-*` namespace via `@theme inline`.

The implementation breaks cleanly into three waves: (1) establish design tokens in `globals.css`, (2) restyle shared layouts and the root layout, (3) restyle individual page components. Because this is purely CSS-class replacement in existing TSX files, no new routes, server actions, or database changes are needed.

**Primary recommendation:** Define all design tokens once in `globals.css` using `@theme`, load Lato via `next/font` with `variable` option, and do a systematic find-and-replace of light-theme Tailwind classes (white, gray, blue) with dark-theme equivalents throughout all 8 targeted files.

### Pattern 1: Design Token Definition in globals.css

**What:** All color and font design tokens live in one `@theme` block at the top of `globals.css`. Tailwind generates utility classes from these tokens automatically.

**When to use:** Once, at the start of this phase — every subsequent change references these tokens.

**Example:**
```css
/* Source: https://tailwindcss.com/docs/theme */
@import "tailwindcss";

@theme inline {
  /* Font — maps next/font CSS variable into Tailwind font-* utilities */
  --font-sans: var(--font-lato);

  /* Colors — hex values supported in Tailwind v4 @theme */
  --color-bg-page:    #0f172a;   /* dark navy page background */
  --color-bg-surface: #1e293b;   /* card / surface */
  --color-border:     #334155;   /* all borders */
  --color-accent:     #10b981;   /* emerald at rest */
  --color-accent-hover: #34d399; /* emerald on hover */
  --color-text-primary: #e2e8f0; /* off-white body text */
  --color-error:      #ef4444;   /* destructive red */
  --color-error-light: #f87171;  /* lighter error */
}

@layer base {
  body {
    background-color: #0f172a;
    color: #e2e8f0;
  }
}
```

**Note on `@theme inline`:** Use `inline` variant (not plain `@theme`) when theme variables reference CSS custom properties (like `var(--font-lato)`). This tells Tailwind to inline the actual value at compile time rather than creating a broken variable chain. For static hex color values, plain `@theme` also works, but `@theme inline` is safe for both cases.

### Pattern 2: Lato Font Loading via next/font

**What:** Load Lato from Google Fonts at build time using `next/font/google`. Lato is a non-variable font — weights must be declared explicitly.

**When to use:** In `src/app/layout.tsx`, replacing the current Geist font setup.

**Example:**
```tsx
// Source: https://nextjs.org/docs/app/getting-started/fonts
import { Lato } from 'next/font/google';

const lato = Lato({
  variable: '--font-lato',   // CSS variable name — referenced in globals.css @theme
  subsets: ['latin'],
  weight: ['400', '700'],    // 400 regular + 700 bold — Claude's discretion recommends these two
  display: 'swap',
});

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" className={lato.variable}>
      <body className="antialiased">
        {children}
      </body>
    </html>
  );
}
```

**Why `variable` (not `className`):** Using `variable` injects the font as a CSS custom property (`--font-lato`) onto the `<html>` element. This lets `globals.css` map it into Tailwind via `@theme inline`. Using `className` directly would bypass the Tailwind token system.

### Anti-Patterns to Avoid

- **Mixing hardcoded hex and token classes:** Don't write `text-[#e2e8f0]` inline when `text-text-primary` is available. Tokens exist — use them.
- **Keeping `rounded-lg` or `rounded-xl`:** The design decision is sharp corners (`rounded-sm` = 2px). Replace every instance of `rounded-lg` and `rounded-xl` in scope.
- **Leaving `bg-white` anywhere in scope:** White backgrounds break the entire dark theme. Every `bg-white` must be replaced.
- **Using color muting for secondary text:** CONTEXT.md explicitly says secondary/helper text stays `#e2e8f0`, just smaller — do NOT use `text-gray-400` or `opacity-50` on labels.
- **Using `shadow-*` on cards:** Design decision is no box shadows. Remove any `shadow-sm`, `shadow-lg` etc.
- **Blue accents:** Current codebase uses `blue-600` everywhere (buttons, links, focus rings). Every blue must become emerald (`accent`/`accent-hover`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Font loading | Manual `<link>` to Google CDN or CSS @font-face | `next/font/google` | Handles preloading, self-hosting, FOUC prevention, type-safe weights |
| Color token management | Inline hex values everywhere | `@theme` in globals.css | Single source of truth; changing one variable updates all usages |
| Focus styles | Custom JS focus handling | Tailwind `focus:` variant + CSS border | Browser handles accessibility; CSS is correct layer |
| Dark overlay (modal) | Custom backdrop implementation | `bg-black/40` (existing pattern) | InviteDialog already uses this — keep it |

**Key insight:** Tailwind v4's `@theme` block eliminates the need for any custom token management system. Define once, use everywhere as utility classes.

## Common Pitfalls

### Pitfall 1: The "or" Divider Background

**What goes wrong:** The `<span className="bg-white px-2">or</span>` inside the horizontal rule divider appears as a white rectangle floating on the dark card.

**Why it happens:** The span background must match the *card* surface (`#1e293b`), not the page background. Developers often only change `bg-white` → `bg-bg-page` but the card sits on top of the page.

**How to avoid:** Change to `bg-bg-surface` to match the card background where the divider sits.

**Warning signs:** White rectangle appearing in the middle of the card between Google button and email form.

### Pitfall 2: Lato Requires Explicit Weights

**What goes wrong:** Passing `weight` as an array with the correct values or not passing it at all causes Next.js to throw a build error: "A non-variable font must specify one or more weights."

**Why it happens:** Lato is not a variable font. Unlike variable fonts (e.g., Inter, Geist), you must explicitly declare which weights to load.

**How to avoid:** Always specify `weight: ['400', '700']` explicitly in the `Lato({...})` call.

**Warning signs:** Build error at `next build` or `next dev` startup mentioning "non-variable font" or "weight required".

### Pitfall 3: @theme inline vs @theme for CSS variable references

**What goes wrong:** Using plain `@theme` (not `@theme inline`) when theme values reference CSS custom properties like `var(--font-lato)` causes Tailwind to generate `font-family: var(--font-sans)` → `var(--font-lato)`, which is a double-variable indirection that sometimes breaks in certain environments.

**Why it happens:** Tailwind v4 plain `@theme` variables become CSS custom properties themselves; if they reference another variable, you get chained variables.

**How to avoid:** Use `@theme inline` for any token values that are themselves CSS variables.

**Warning signs:** Font not applying in production even though it works in dev; browser DevTools showing `font-family: var(--font-sans)` but no actual font name.

### Pitfall 4: AuthLayout Missing Dark Background

**What goes wrong:** The `(auth)/layout.tsx` currently wraps children in `bg-gray-50`. If you restyle individual page cards but forget the layout wrapper, the gray background shows behind the dark card.

**Why it happens:** Auth pages have two layers — the layout wrapper (full-page bg) and the card (page component). Both must change.

**How to avoid:** Check `(auth)/layout.tsx` — change `bg-gray-50` to `bg-bg-page min-h-screen`.

**Warning signs:** Gray perimeter visible around dark auth cards.

### Pitfall 5: Join Page Has Inline Dark Background Styles

**What goes wrong:** `join/[token]/page.tsx` renders its own full-page wrappers (`min-h-screen flex items-center justify-center bg-gray-50 p-4`) on each conditional render path — it does NOT use a shared layout. Each `return (...)` branch must be individually restyled.

**Why it happens:** Join page is outside the `(auth)` and `(trainer)`/`(trainee)` route groups — it has no layout wrapper. There are 4 conditional render paths (invalid invite, conflict, error, success redirect).

**How to avoid:** Update all 3 error-state return branches in `join/[token]/page.tsx` independently. Check each `return` statement.

**Warning signs:** Dark theme everywhere except the join page.

### Pitfall 6: Trainee Avatar on Trainer Roster Page

**What goes wrong:** Trainer home page shows trainee initials in `bg-blue-100 text-blue-700` (avatar circles). On a dark background, this blue/light-blue is jarring and inconsistent.

**Why it happens:** The avatar styling was added in Phase 1 with the light theme in mind.

**How to avoid:** Change avatar to `bg-accent/20 text-accent` (emerald tint background with emerald text) to match the design system.

**Warning signs:** Light blue circles floating on dark cards.
