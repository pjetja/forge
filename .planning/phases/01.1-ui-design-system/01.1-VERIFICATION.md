#### 1. Error Message Color Acceptability

**Test:** Navigate to `http://localhost:3000/login`. Enter a valid email and a wrong password. Submit the form.
**Expected:** An error message appears in a dark semi-transparent red box. Confirm the white text on the dark red background is readable and visually acceptable.
**Why human:** The plan specifies `text-error` (red text) but the implementation uses `text-white`. The automated check cannot judge visual readability or whether the divergence is acceptable.

#### 2. Lato Font Load Confirmation

**Test:** Open `http://localhost:3000/login` in a browser. Open DevTools > Network tab. Filter by "Font" or "woff2". Reload the page.
**Expected:** A `.woff2` file is fetched from `/_next/static/media/` (the Lato font). The computed `font-family` on `<body>` in DevTools Elements panel shows "Lato".
**Why human:** Font loading and FOUC prevention cannot be verified via static code analysis.

#### 3. Auth Pages Full Visual Sweep

**Test:** Visit each auth page: `/login`, `/signup/trainer`, `/signup/trainee`, `/verify-email`.
**Expected:** Each page shows a dark navy full-page background (no gray or white perimeter), a centered dark surface card, off-white text, emerald links and buttons. No legacy light elements visible.
**Why human:** Visual rendering requires browser inspection.

#### 4. App Shell Pages and InviteDialog
