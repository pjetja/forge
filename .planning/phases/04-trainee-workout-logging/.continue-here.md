<decisions_made>

- "Week" is a META concept: every N completed sessions = 1 week (N = workouts_per_week). No calendar dates.
- Plan completion: computed as `allCompleted.length >= week_count * workouts_per_week`. Page proactively marks DB on load if mismatch.
- Admin client pattern: used everywhere trainees need to read/write data beyond their RLS permissions (exercises table read, assigned_plans status update).
- FK-hint joins in Supabase broken: no FK from trainer_trainee_connections.trainee_auth_uid to public.users.auth_uid exists. All such joins replaced with two separate queries.
- 0kg is valid weight (bodyweight exercises). Fixed `parseFloat(v) || null` → explicit empty check.
- Start button always visible even after workout is Done this week — trainee can redo. Only blocked if another session is in_progress.
</decisions_made>

<blockers>

- exercises table RLS is trainer-only. Trainee pages must use admin client to read exercise names/video_url. A proper fix would be a new RLS policy (SQL provided in conversation) but not yet applied to DB.
- A migration SQL snippet was provided for trainee exercise access:
  ```sql
  CREATE POLICY "trainee_reads_exercises_in_assigned_plans" ON exercises
    FOR SELECT USING (
      id IN (
        SELECT ase.exercise_id FROM assigned_schema_exercises ase
        JOIN assigned_schemas asc ON asc.id = ase.assigned_schema_id
        JOIN assigned_plans ap ON ap.id = asc.assigned_plan_id
        WHERE ap.trainee_auth_uid = (SELECT auth.uid())
      )
    );
  ```
